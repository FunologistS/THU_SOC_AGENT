# 流程优化建议：稳健生成 + 学术可用质量

在「不超时/不崩溃」和「学术研究可用」两个目标下，按**稳健性**与**质量**分类的优化项与优先级。

---

## 一、稳健性（少超时、不崩溃）

| 优先级 | 优化项 | 说明 | 状态/建议 |
|--------|--------|------|-----------|
| **P0** | 一键综述前自动 compact_report | 无 chunks 时先切块再改写，避免整篇一次送模型 | ✅ 已实现（run API） |
| **P1** | 文献简报「每聚类最多 N 篇」 | 限制每聚类送进 LLM 的论文数，避免单次请求几百篇超时 | ✅ 已实现：API 默认传 `--max-papers-per-cluster 40`，前端可选传 `conceptMaxPapersPerCluster` |
| **P2** | 合并阶段默认预算略放宽 | 默认 MERGE_MAX_CHARS=22000，减少「必须分层合并」的情况，单次合并更稳；遇 504/520 再改为 15000 | ✅ 已实现：默认 22000 |
| **P3** | 合并 504/520 时自动重试降预算 | 检测到 504/520 时自动用 MERGE_MAX_CHARS=12000 再试一次（或仅分层） | ✅ 已实现：merge-only 与 chunk 模式均自动重试一次 |
| **P4** | 文献简报聚类内分批 | 单聚类论文数 > N 时，分多批调 LLM 再合并该聚类段落 | ✅ 已实现：concept_synthesize_gpt/glm 均支持分批，附录仅最后一批请求 |

---

## 二、质量（学术可用）

| 优先级 | 优化项 | 说明 | 建议 |
|--------|--------|------|------|
| **Q1** | 分层合并后建议做衔接优化 | 当本次走了「分层合并」时，自动开启一次段落衔接与术语统一（或 UI 建议用户勾选） | ✅ 已实现：一键综述确认页文案「主题数 ≥ 8 时建议勾选」 |
| **Q2** | 风格样本上限略放宽 | MAX_STYLE_CHARS 默认 2000 → 3000，学术写作时让模型看到更多样本文风 | ✅ 已实现：默认 3000 |
| **Q3** | 综述产出简短质量提示 | 在 06_review 同目录写一份 `review_latest_说明.txt`：建议人工核对引用、主题覆盖、衔接 | ✅ 已实现：合并成功后自动写入 |
| **Q4** | 主题覆盖自检（可选） | 合并后检查输出中 `##` 数量是否接近输入主题数，若明显少则打 warning 或写入说明 | ✅ 已实现：自检结果写入说明文件并打 warning |

---

## 三、已落地的默认行为（当前）

- **文献简报**：API 默认 `--max-papers-per-cluster 40`；前端文献简报弹窗有「每聚类最多使用篇数」下拉（20/30/40/50/100），单聚类超过该数会分批调 LLM 再合并该聚类段落。
- **一键综述**：合并总长默认 22000 字；超长时自动分层合并（不截断单主题内容）；合并阶段 504/520 或超时会自动以 MERGE_MAX_CHARS=12000 重试一次；成功后写入 `review_latest_说明.txt` 并做主题覆盖自检；确认页文案提示「主题数 ≥ 8 时建议勾选段落衔接优化」。
- **风格样本**：MAX_STYLE_CHARS 默认 3000。

---

## 四、建议使用习惯（学术场景）

1. 文献简报：聚类多或单聚类论文多时，在 UI 将「每聚类最多篇数」设为 30～50，既稳又保留足够信息量。
2. 一键综述：主题数 ≥ 8 时建议勾选「段落衔接优化」；若合并报 504/520，用 `MERGE_MAX_CHARS=15000 node ... --merge-only` 重跑合并。
3. 定稿前：人工核对 06_review 中引用是否完整、各主题是否都有对应段落、衔接是否自然。
