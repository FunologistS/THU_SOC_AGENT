# 文献简报与一键综述稳健性改进说明

本文档说明：当文献数量较多时，**文献简报**和**一键综述**容易因单次请求过大而超时；当前已有的「分块/限流」机制为何没有稳定生效；以及按优先级排列的**解决方案建议**。

---

## 一、现状与问题

### 1. 现象

- **文献简报**：论文多时，运行到一半或末尾报超时（520/504/timeout）。
- **一键综述**：报告长时，整篇一次送模型，容易超时或网关错误。

### 2. 根因简述

系统里其实已经有「分块、限流」的设计，但在**主流程里没有被稳定触发**：

| 环节 | 已有能力 | 当前为何没生效 |
|------|----------|----------------|
| **文献简报** (concept_synthesize) | 支持 `--max-papers-per-cluster N`，每个聚类只取前 N 篇再送 LLM。 | 默认 N=999，且 UI/API 从未传该参数 → 等于不限制 → 单次请求可能几百篇 → 易超时。 |
| **报告切块** (compact_report) | 按 `##` 把 `05_report/report_latest.md` 切成多个 chunk 文件。 | 只在「上传写作样例并运行」时才会先跑 compact_report；用户若直接点「一键综述」且选「参考既有风格」，不会跑 compact_report → 没有 chunks。 |
| **一键综述** (writing_under_style) | 若存在 `05_report/chunks/`，会按块改写再合并（chunk 模式）。 | 多数情况下 chunks 不存在（见上）→ 退化为「整篇 report 一次送模型」→ 长报告易超时。 |

结论：**三种 chunk 相关能力都未在主流程中稳定生效**，导致「文献多/报告长」时稳健性不足。

---

## 二、解决方案建议（按优先级）

### P0：一键综述前自动跑 compact_report

| 项目 | 说明 |
|------|------|
| **建议** | 只要用户点「一键综述」，在调用 writing_under_style 之前，若存在 `05_report/report_latest.md` 且 `05_report/chunks/` 不存在或为空，则**先执行一次** `2_compact_report.mjs <topic>`，再执行 writing_under_style。 |
| **实现位置** | 二选一：① 在 **run API**（`jobType === "writing_under_style"` 时）里先判断并调用 compact_report；② 或在 **3_writing_under_style.mjs** 入口处判断，若无 chunks 则先 spawn compact_report 再继续。 |
| **预期效果** | 一键综述稳定走 **chunk 模式**，长报告不再整篇一次送模型，超时/520 明显减少。 |
| **工作量** | 小（一段判断 + 一次脚本调用）。 |

---

### P1：文献简报「每聚类最多 N 篇」接入 API + UI

| 项目 | 说明 |
|------|------|
| **建议** | 后端：run 接口在处理 `concept_synthesize` 时，接收并传入 `--max-papers-per-cluster N`（若前端未传则用合理默认值，如 30 或 50）。前端：在文献简报的配置区域增加「每聚类最多使用篇数」选项（如 20/30/50/100/不限制）。 |
| **预期效果** | 文献多时单次请求体量受控，超时率明显下降。 |
| **工作量** | 小（后端拼 1 个参数 + 前端 1 个控件）。 |

---

### P2：文档更新

| 项目 | 说明 |
|------|------|
| **建议** | 在《常见错误与处理》或相关 SKILL 中写明：① 一键综述在「有 chunks 时」会按块改写（更稳），若采纳 P0 则注明「点一键综述时会自动先切块」；② 文献简报在「每聚类篇数较多」时可能超时，建议使用「每聚类最多 N 篇」（若采纳 P1 则注明默认值及在 UI 何处设置）。 |
| **预期效果** | 行为可预期，排查问题时更容易理解。 |
| **工作量** | 很小。 |

---

### P3：文献简报「聚类内分批调用」（可选）

| 项目 | 说明 |
|------|------|
| **建议** | 在 concept_synthesize 脚本中，对每个聚类若论文数 > maxPapersPerCluster，不只取前 N 篇，而是**分多批**（每批 N 篇），每批调一次 LLM 得到一段简报，最后将该聚类的多段合并。 |
| **预期效果** | 单聚类极多篇时仍能稳定跑完，单次请求体量固定。 |
| **工作量** | 中（需改 concept_synthesize 的循环与合并逻辑）。 |

---

## 三、建议实施顺序

| 优先级 | 建议 | 预期效果 | 工作量 |
|--------|------|----------|--------|
| **P0** | 一键综述前自动跑 compact_report | 一键综述稳定走 chunk 模式，长报告不再整篇送模型 | 小 |
| **P1** | 文献简报 max-papers-per-cluster 接入 API + UI | 文献多时单次请求变小，超时明显减少 | 小 |
| **P2** | 文档更新 | 行为可预期、排查容易 | 很小 |
| **P3** | 聚类内分批调用 | 单聚类极多篇时更稳 | 中 |

建议先做 **P0**，再做 **P1**；P2 可与 P0/P1 同步或紧随其后；P3 视实际超时情况再决定是否做。

---

## 四、为什么有时文献简报主题数比一键综述多？（主题数不一致时）

### 数据流简述

| 步骤 | 产出 | 主题/块数来源 |
|------|------|----------------|
| **荟萃分析** (synthesize) | `04_meta/meta_clusters_*.md`、`briefing_*.md` 等 | 聚类数 k（或自动选 k）→ 有 N 个 Cluster |
| **文献简报** (concept_synthesize) | `05_report/report_*.md`、`report_latest.md` | 读当前 04_meta 的 meta_clusters，**每个聚类**生成 report 里的一节（`## 一、…`、`## 二、…` …）→ **主题数 = 当时跑的聚类数** |
| **一键综述** (writing_under_style) | `06_review/review_*.md` | 输入来自 **05_report**：有 `chunks/` 则按现有 chunk 数改写；没有则用 `compact_report` 按 `## ` 把 report 切块 → **块数 = report 里的一级标题数** |

因此：**一键综述实际改写的主题数 = 当前 05_report 里那份 report 的 `## ` 节数**（或已有 chunks 的文件数），和 04_meta 里「现在」的聚类数可以不一致。

若 **report 与 chunks 里已经是 10 个主题**，逐块改写会产出 10 段；但**合并阶段**可能只把其中一部分交给模型，导致最终综述里主题变少（见下）。

### 原因一：05_report 是旧版（主题数本来就少）

- 你先跑过**文献简报**（当时 04_meta 是 **7 个聚类**）→ 生成了 7 个主题的 report/chunks。
- 之后重新跑了**荟萃分析**并改了 k → **04_meta 现在有 10 个聚类**；**05_report 没有重新生成**，所以一键综述仍只改写/合并那 7 块。

### 原因二：合并阶段被截断（report/chunks 有 10 个，但最终综述只有 7 个）

一键综述的**合并**在 `3_writing_under_style.mjs` 里完成：先把所有 `chunks_styled/*.md` 拼成一篇草稿，再按 **MERGE_MAX_CHARS**（默认 15000）做「总 prompt 上限」裁剪，只把前 N 个字符发给模型做「整合为一篇连贯综述」。  
若 10 个主题的改写合起来超过该预算（风格样本 + 前缀/后缀会占掉一部分，草稿实际预算约 1.2 万字符），**后面的 chunk 会被截掉**，模型只收到前 7 个主题左右的内容，最终综述就只包含那部分。

- **脚本位置**：`.claude/skills/paper-writing/scripts/3_writing_under_style.mjs`
- **相关逻辑**：`fitDraftToPromptBudget()` 用 `MERGE_MAX_CHARS` 限制「风格 + 前缀 + 草稿 + 后缀」总长，草稿超出部分用 `truncateByCharsMarkdownSafe()` 截断。

### 解决办法

**当 05_report 是旧版（原因一）时：**

1. 重新跑一次「文献简报」，用当前 04_meta 生成新 report。  
2. 若 05_report 下已有 chunks 且是旧份，删掉 `outputs/<主题>/05_report/chunks/` 再跑一键综述，让 compact_report 按新 report 重新切块。  
3. 一键综述的「输入文档」选默认或刚生成的那份 report。

**当合并被截断（原因二）时：**

- 脚本已优化为**分层合并**：草稿超长时按 `##` 分节，将节分成多组（每组总长不超过单次请求预算），**每组完整内容**先各自合并一次，再把这几段合并结果做一次总合并。这样**不截断任一主题的完整内容**，只增加合并阶段的 API 调用次数（例如 10 主题拆成 2 组则共 3 次调用：组1 合并、组2 合并、总合并）。若只有一节且该节超长，才退化为按节均分字数压缩。可调大 `MERGE_MAX_CHARS` 减少分组数；若只重跑合并：  
  `MERGE_MAX_CHARS=25000 node .claude/skills/paper-writing/scripts/3_writing_under_style.mjs <topic> --merge-only`
