# 常见错误与处理参考

本文档面向使用本系统的用户（如导师、合作者），说明使用过程中可能遇到的典型错误、原因以及可行的处理办法。遇到报错时，可先根据**错误现象**在本页查找对应条目。

---

## 一、检索文献时

### 1. HTTP 429 / Rate limit exceeded / Insufficient budget

**现象**  
开始检索或检索到一半时失败，日志里出现类似：

- `HTTP 429 Too Many Requests`
- `Rate limit exceeded`
- `Insufficient budget. This request costs $0.001 but you only have $0.0009 remaining. Resets at midnight UTC.`

**原因**  
检索使用的是 **OpenAlex** 的接口。OpenAlex 目前采用**按用量计费**（usage-based pricing），而不是“每天 100k 次调用”的旧说法：

- 每个 API Key 每天有 **$1 免费额度**（午夜 UTC 重置）。
- 不同操作扣费不同：**按关键词/全文检索（search）** 每次请求约 **$0.001**，按条件列表筛选（list+filter）每次约 $0.0001，按 ID/DOI 单条查询免费。
- 本系统的「按期刊+主题检索」会多次调用 **search** 接口（每个期刊、每页结果都可能是一次请求），所以**点一次“开始检索”可能对应几十到上千次 API 请求**。若选了很多期刊或结果很多，很容易在一天内把 $1 额度用光，再发一次请求就会报“Insufficient budget”。

**为什么官网写的是“Free 100k/day”？**  
官网的 [Pricing](https://openalex.org/pricing) 页主要描述的是**订阅档位**（Free / Premium / Institutional）。**实际 API** 已改为“每天 $1 免费额度 + 按请求类型扣费”，详见 [OpenAlex 官方说明](https://blog.openalex.org/openalex-api-new-features-and-usage-based-pricing/) 与 [API 认证与计费](https://docs.openalex.org/how-to-use-the-api/rate-limits-and-authentication)。在 $1 额度下，大约可做约 **1,000 次 search** 或 10,000 次 list+filter，超出后就会 429。

**怎么办**  
1. **等额度重置**：免费额度在 **午夜 UTC**（北京时间早上 8 点）重置，第二天再试。  
2. **务必配置 OpenAlex API Key**：在 [openalex.org/settings/api](https://openalex.org/settings/api) 注册并获取 API Key，在系统**设置**里填入 `OPENALEX_API_KEY`。未配置时匿名额度更小，更容易 429。  
3. **减少单次检索量**：少选一些期刊、缩短年份范围、或收窄检索词，可减少请求次数，在 $1/天内完成。  
4. **充值**：若经常需要大批量检索，可到 [openalex.org/pricing](https://openalex.org/pricing) 购买预付费用量（prepaid usage）。

### 2. Default journals.yml not found / 未找到期刊列表

**现象**  
点击「开始检索」时提示找不到默认期刊列表，或报错里出现 `journals.yml` 相关路径。

**原因**  
系统需要一份期刊清单（journals.yml）才能按期刊检索。该文件通常由项目维护者放在指定位置；若被移动、删除或项目未正确部署，就会找不到。

**怎么办**  
1. 确认项目是从完整代码库打开的，且没有删除过 `.claude/skills/journal-catalog/references/system/` 下的文件。  
2. 联系项目维护者（如你的学生）确认 `journals.yml` 的正确路径，并恢复或重新生成该文件。

## 二、荟萃分析 / 文献简报时

### 3. Input not found / summaries_latest.md 或 04_meta 未找到

**现象**  
运行「荟萃分析」或「文献简报」时失败，日志里出现：

- `Input not found: .../03_summaries/summaries_latest.md`
- `Meta clusters not found` / `Briefing not found` / `Summaries not found`

**原因**  
这些步骤有**先后顺序**依赖：  
- **荟萃分析**需要先有「论文摘要」结果（即 `03_summaries/summaries_latest.md`）。  
- **文献简报**需要先跑完「荟萃分析」，即存在 `04_meta` 下的聚类、简报等文件。

若还没做过上一步，或上一步失败、未生成对应文件，就会出现“未找到”。

**怎么办**  
1. **荟萃分析报错**：先完成「检索」和「论文摘要」两步，确认该主题下已生成 `outputs/<主题名>/03_summaries/summaries_latest.md`，再点荟萃分析。  
2. **文献简报报错**：先完成「荟萃分析」，确认 `outputs/<主题名>/04_meta/` 下已有相应文件，再点文献简报。  
3. 若文件本来就有却仍报错：检查是否选错了**主题（topic）**，确保当前主题与文件所在目录一致。

### 4. 520 / 504 / 超时 / 网关错误（写作或文献简报时）

**现象**  
运行「文献简报」或「一键综述」时，长时间无响应或报错中出现：

- `520`、`504`、`Gateway`、`timeout`、`ETIMEDOUT` 等。

**原因**  
调用的大模型接口（如 OpenAI、智谱）有单次请求大小或时长限制。当摘要/报告内容很多时，一次发送的内容过长，容易触发网关限流或超时。

**怎么办**  
1. **文献简报**：可联系维护者在运行「文献简报」时加上“每聚类最多多少篇”的限制（若系统支持），减少单次请求体量。  
2. **一键综述**：  
   - 先让系统对报告做「分块」（若流程中有“上传并写作”或“转录并写作”，通常会先分块）；或由维护者先运行一次「分块报告」脚本，再跑一键综述。  
   - 若仍超时，可适当缩短检索范围（如少选期刊、缩短年份），使摘要和报告体量变小后再试。  
3. 过一段时间再试，有时是接口侧临时限流。

## 三、一键综述 / 写作时

### 5. OpenAI API key / 智谱 API key 未配置或无效

**现象**  
运行「一键综述」或与写作相关的步骤时失败，提示：

- 缺少 API Key、无效 Key、认证失败、401/403 等。

**原因**  
综述改写和部分摘要功能依赖**第三方大模型接口**（如 OpenAI、智谱），需要在系统**设置**中配置对应的 API Key。未配置或 Key 过期、填错，都会报错。

**怎么办**  
1. 在系统**设置**（或项目中的 `.env.local`）里填写：  
   - 使用 GPT 时：`OPENAI_API_KEY`，以及如需要可填 `OPENAI_BASE_URL`。  
   - 使用智谱时：`ZHIPU_API_KEY`。  
2. 确认 Key 未过期、未多复制空格；若不确定，可到对应平台（OpenAI 或智谱）后台查看 Key 状态或重新生成。  
3. 若通过学校/机构代理访问，需确认 `OPENAI_BASE_URL` 等是否指向了正确的代理地址。

---

### 6. 报告/输入文件未找到（compact_report 或 writing 报错）

**现象**  
运行与「一键综述」相关的流程时，提示输入文件或 report 不存在，例如：

- `input not found: .../05_report/report_latest.md`
- 未找到 chunks 目录 / 未找到 .md chunk 文件

**原因**  
一键综述的输入是「文献简报」生成的报告（`05_report/report_latest.md`），或由该报告分块得到的多个 chunk 文件。若还没跑「文献简报」，或报告生成失败，就不会有这些文件。

**怎么办**  
1. 按顺序先完成：检索 → 论文摘要 → 荟萃分析 → **文献简报**，确认 `outputs/<主题名>/05_report/` 下已有 `report_latest.md`（或 chunks）。  
2. 若维护者单独跑过「分块」脚本，确认 `05_report/chunks/` 目录存在且内有 `.md` 文件。  
3. 检查当前选择的**主题名**是否与产出文件所在目录一致。

## 四、其他

### 7. 网络连接失败 / ECONNREFUSED / 无法访问外网

**现象**  
任意步骤报错中出现：`ECONNREFUSED`、`ENOTFOUND`、`network`、`fetch failed` 等。

**原因**  
本系统需要访问外部服务（OpenAlex、OpenAI、智谱等）。若本机或所在网络无法访问外网、或防火墙/代理拦截，就会出现连接失败。

**怎么办**  
1. 确认本机可以正常上网，并能访问 openalex.org、OpenAI/智谱的 API 域名。  
2. 若在校园网/公司网络，确认是否需要配置代理；若需要，由维护者在环境变量中配置相应的 `BASE_URL` 或代理设置。  
3. 临时换一个网络（如手机热点）试一次，以排除当前网络限制。

## 五、界面 / 前端

### 9. GET /_next/static/chunks/main-app.js 或 app-pages-internals.js 返回 404

**现象**  
终端里反复出现：

- `GET /_next/static/chunks/main-app.js?v=... 404`
- `GET /_next/static/chunks/app-pages-internals.js 404`
- 有时还伴随 `GET /_next/static/css/app/layout.css?v=... 404`

页面可能白屏、按钮点击无反应，或样式错乱。

**原因**  
这是 Next.js 开发环境下**静态资源路径与缓存不一致**导致的：页面 HTML 里引用的脚本/样式文件名（或版本号）与当前编译产物不一致，请求时就会 404。常见触发情况：

- 开发服务器中途重启或多次热更新后，旧的 chunk 名已失效，浏览器仍按旧地址请求。
- 项目路径中含有**空格或中文**（如 `Mobile Documents`、`孙百承`）时，个别环境下 Next 对静态文件的解析会异常，导致 404。

**怎么办**  
1. **先清缓存并重启开发服务**（在项目 `ui/app` 目录下执行）：
   ```bash
   rm -rf .next
   npm run dev
   ```
   然后**强制刷新**浏览器（Ctrl+Shift+R 或 Cmd+Shift+R），必要时清除站点对该地址的缓存后再打开页面。  
2. 若仍 404：尝试把项目**复制到不含空格和中文的路径**（例如 `~/THU_SOC_AGENT`）再在该路径下执行 `npm run dev`，看是否恢复正常。  
3. 若仅偶发 404 且刷新后页面能正常用，可忽略；若每次打开都 404 且页面不可用，把终端里 404 的完整 URL 和 `npm run dev` 的启动输出一并发给维护者排查。

### 10. 如何把错误信息提供给维护者

若以上都无法解决，可把下面信息发给项目维护者（如你的学生），便于排查：

1. **在做什么**：例如「点击了开始检索」「运行了一键综述」。  
2. **主题名**：当前选择的 topic/主题。  
3. **完整报错**：日志或界面中红色的错误全文（可复制粘贴），尤其是包含 `Error:`、`HTTP`、文件路径的那几行。  
4. **是否刚改过设置**：例如刚填了或改了 API Key、换了网络。
